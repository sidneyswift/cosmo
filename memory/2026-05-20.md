## QA Judge Built (1:30 AM, May 21)

### What Sid Asked
Content system isn't smart enough. Three requirements:
1. Copywriting skill should be used on every write
2. General safety rules (no customer names, no internal numbers) enforced by an LLM judge
3. Judge learns from Sid's feedback, rewrites failures, and gets smarter over time

### What I Built: `scripts/qa-judge.js`

**Architecture:**
```
Writer (Sonnet) → Critic (GPT-4o score) → QA Judge (GPT-4o eval + rewrite loop) → Dashboard → Sid's feedback → Judge learns
```

**Judge evaluates 4 dimensions:**
1. Safety (hard fails): customer names, internal numbers, Flex references, fabricated data
2. Copywriting quality (soft fails): em dashes, reframe shapes, engagement bait, hype
3. Voice & originality: does it sound like Sid, does it have a unique POV
4. Platform fit: right length, hook quality, structure

**Rewrite loop:**
- Verdict: pass / rewrite / reject
- On rewrite: sends specific notes to a rewriter agent (Sonnet), gets revision, re-judges
- Max 2 rewrite rounds before rejection
- Safety violations = automatic reject (can't rewrite away)

**Learning system:**
- `sync-from-supabase.js` detects Sid's feedback (status=revising or rejected with notes)
- Calls `qa-judge.js --learn` to extract a general rule from the feedback
- Saves to `data/guardrails.json → learned_rules`
- Judge prompt includes all learned rules with highest priority

**Integration:**
- Nightly engine: step 8 replaced selfReviewDrafts (regex) with judgeAllReady (LLM)
- Cron daemon: QA judge runs every 2 minutes between regenerate and auto-slides
- Full copywriting skill + anti-slop.md + formats.md loaded into judge and rewriter prompts

**Files:**
- `scripts/qa-judge.js` — the judge (new)
- `data/guardrails.json` — safety rules + learned rules (existing, extended)
- `scripts/nightly-engine.js` — wired judge into step 8
- `scripts/cron-daemon.js` — added qa-judge to 2-min loop
- `scripts/sync-from-supabase.js` — added feedback learning

## Content Pipeline Sync Fix (5:00 PM)

### Root Cause
Supabase sync broken since May 18. Two issues:
1. **Cron daemon didn't inject Supabase env vars** — `cron-daemon.js` ran child scripts with `process.env` (no Supabase vars). The nightly engine had its own `getSupabaseEnv()` but that only helped scripts run BY the nightly engine, not the every-2-min auto-slides run directly by the cron daemon.
2. **auto-slides.js depended entirely on Supabase** — read `media_review` drafts from Supabase, generated thumbnails, uploaded to Supabase storage, updated Supabase records. If Supabase was unreachable, nothing happened.

### What Was Broken
- 20+ ready drafts sitting with no thumbnails since May 18
- Nightly engine ran May 19 but all sync steps failed → no thumbnails → no scheduling → no LinkedIn posts
- auto-slides running every 2 min but silently exiting (no Supabase key)

### Fixes Applied
1. **cron-daemon.js** — Added `loadSupabaseEnv()` at startup that sources `~/.openclaw/credentials/supabase-env.sh` and injects vars into `process.env`. All child scripts now inherit them.
2. **auto-slides.js** — Rewired to read from LOCAL SQLite as primary source (not Supabase). Supabase upload is best-effort. Falls back to local file paths if upload fails. Also:
   - Batch size: max 3 per run (was unlimited — 20 at once triggered rate limits)
   - 5s delay between generations
   - Retry logic (2 retries) for transient safety filter rejections
   - Updates local SQLite `media_urls` directly (no Supabase round-trip needed)
3. **nightly-engine.js** — Updated thumbnail check to use local DB status instead of `synced_to_dashboard` flag. Fixed `verifyAndMoveReady` to exclude empty-string media_urls.

### Result
- Cron daemon restarted with Supabase env ✅
- 246 ideas synced to Supabase ✅  
- auto-slides now generating thumbnails from local DB — processing 3 per 2-min cycle
- 4 approved drafts auto-published to LinkedIn via sync-from-supabase
- Pipeline is self-healing: even if Supabase goes down again, thumbnails still generate locally

### Architecture Lesson
Local SQLite should always be the source of truth for the content engine. Supabase is a mirror for the dashboard, not a dependency. This fix makes the pipeline Supabase-optional for core operations (generate, review, thumbnail, schedule).

2. No safety guardrails (customer names, financials leaking)
3. No LLM-as-judge that learns from his feedback and rewrites automatically

### What I Built

**New: `scripts/qa-judge.js`** — LLM-based quality gate replacing the regex selfReviewDrafts().
- 5 evaluation dimensions: copywriting rules, safety guardrails, tone/humanness, style calibration, learned principles
- **Fake story detector** — flags fabricated narrative arcs and invented specifics
- **AI voice detector** — catches over-explaining, corporate storytelling cadence
- **Style calibration** — compares drafts against Sid's actual top-performing posts (hardcoded reference set)
- **Observation framing** — prefers "here's what I'm seeing" over "when we started Recoup..."
- Rewrite loop: if judge fails a draft, sends specific notes to rewriter, re-judges (max 2 rounds)
- Safety pre-check (regex, no LLM) for instant rejects on banned names/patterns

**New: `data/guardrails.json`** — Safety rules the judge loads:
- Banned names: Rostrum, APG, Flex Seal, Phil Swift, DRTV, swiftdrtv
- Banned patterns: customer counts, financial metrics, funding info
- Self-updating: when Sid's feedback implies new safety rules, `updateGuardrails()` adds them

**Updated: `scripts/nightly-engine.js`** — Replaced selfReviewDrafts() with `await judgeAllReady()`. Old function kept but commented as legacy.

**Updated: `scripts/regenerate.js`** — Added guardrail learning after principle extraction. When Sid gives dashboard feedback, both principles.json AND guardrails.json get updated.

### New Pipeline Flow
```
Writer → Critic (score) → QA Judge (5-dim checklist + rewrite loop) → media_review → thumbnail → schedule
                                                                            ↑
                                         Sid's dashboard feedback → extractPrinciples() + updateGuardrails()
```

### Key Design Decisions
- Judge uses GPT-4o (same as critic) for evaluation

## Content Pipeline: Dedup System Built (11:55 PM)

### Problem
Brain-first generator mined Sid's meetings and created 3 variations of "we only have one enterprise customer", 2 versions of "Google Drive permissions", 2 versions of "celebrity background", etc. Pulse also didn't catch thematic dupes from different URLs.

### Solution: `scripts/dedup.js`

New semantic deduplication module with two layers:
1. **Fast keyword check** (no LLM) — extracts keywords, computes Jaccard similarity. Threshold 0.5 = likely dupe, 0.7 = auto-skip
2. **LLM semantic check** (gpt-4o-mini, cheap) — for borderline cases (0.3-0.7 keyword similarity), asks LLM to compare the new idea against top candidates

Exports:
- `isDuplicate(title, summary)` — single idea check
- `batchDedup(ideas[])` — filters an array, also dedupes WITHIN the batch
- `rebuildIndex()` — refreshes the in-memory index

### Wired Into:
1. **`nightly-engine.js`** — new `dedupPass()` runs before self-triage, catches dupes in unreviewed ideas
2. **`brain-first-generate.js`** — `batchDedup()` filters moments before generating posts
3. **CLI** — `node scripts/dedup.js --scan` to check current pipeline, `--stats` for index info

### Verified
- Scan found 18 duplicate clusters in the broader pipeline (most already in skip status)
- All files pass syntax + import checks
- Index: 307 ideas, avg 26.5 keywords per idea
