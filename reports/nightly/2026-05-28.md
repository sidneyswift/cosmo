## Phase 3: What I Built

### 1. approve.js Thumbnail Validation Gate

**The bug (reported May 21, again May 27):** approve.js published ALL approved drafts regardless of whether they had thumbnails. Legacy drafts with stale/missing media went out bare.

**Root cause #1:** No validation gate — any approved draft gets published immediately.

**Root cause #2:** `media_urls` handling was broken in two ways:
- auto-slides stores thumbnails in Supabase and sets `media_urls` to a Supabase URL array
- approve.js only checked for local files in `data/media/slides/draft-{id}/` (wrong directory naming)
- Supabase adapter returns `media_urls` as a pre-parsed array, but approve.js assumed it was always a JSON string and called `JSON.parse()` on it, which threw

**Fix (3 changes to approve.js):**

1. **Thumbnail gate before publishing:** Before scheduling any draft, checks for either:
   - Local slide PNGs in `data/media/slides/draft-{id}/`
   - Valid `media_urls` (Supabase URLs or local paths — NOT stale Postiz URLs)
   - If neither exists → sends draft back to 'ready' status with a log message

2. **Supabase URL pipeline in uploadMedia:** When `media_urls` contains Supabase storage URLs:
   - Downloads thumbnail to temp file via curl
   - Uploads to Postiz via `postiz upload`
   - Cleans up temp file
   - Returns Postiz CDN URL for the post

3. **Polymorphic media_urls parsing:** Handles both pre-parsed arrays (Supabase adapter) and JSON strings (SQLite) with `Array.isArray(raw) ? raw : JSON.parse(raw)`

**Test results:**

- ✅ New draft #922 with Supabase thumbnail: downloaded → uploaded to Postiz → published with image
- ✅ The 3 legacy drafts that were previously published bare now have valid Supabase thumbnails and would publish correctly if re-approved
