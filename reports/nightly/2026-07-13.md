## Phase 3: What I Built

### Resilient db-adapter with SQLite fallback

**Modified files:**
- `scripts/lib/supabase-client.js` — `getSupabaseClient()` returns `null` instead of throwing when no credentials found. Added `isSupabaseAvailable()` export for clean checking.
- `scripts/lib/db-adapter.js` — Complete rewrite with dual-mode architecture:

**How it works:**
1. On first call, checks `isSupabaseAvailable()`
2. If Supabase creds exist → uses Supabase (identical to previous behavior)
3. If no creds → logs one warning, falls back to SQLite for all operations
4. Every method (20+ methods across drafts and ideas) has a SQLite implementation
5. SQLite implementations use the existing `db.js` better-sqlite3 connection
6. Performance, voice, and topic multiplier operations always use SQLite (unchanged)

**SQLite fallback covers:**
- `getDrafts`, `getDraft`, `getDraftByLocalId`, `insertDraft`, `updateDraft`
- `updateDraftByLocalId`, `deleteDrafts`, `countDraftsByStatus`, `countDraftsWhere`
- `draftExistsForIdea`, `getDraftsNeedingMedia`, `getDraftsForSlackCheck`, `getPublishedDrafts`
- `getIdeas`, `getIdea`, `updateIdea`, `insertIdea`, `countIdeas`, `updateIdeasWhere`

**New diagnostic:**
- `dbAdapter.mode` → returns `'supabase'` or `'sqlite'`

**Commit:** `990d537` on content-engine main
